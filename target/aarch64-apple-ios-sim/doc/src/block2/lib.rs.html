1//! # Apple's C language extension of blocks
2//!
3//! C Blocks are functions which capture their environments, i.e. the
4//! C-equivalent of Rust's [`Fn`] closures. As they were originally developed
5//! by Apple, they're often used in Objective-C code. This crate provides
6//! capabilities to create, manage and invoke these blocks, in an ergonomic,
7//! "Rust-centric" fashion.
8//!
9//! At a high level, this crate contains four types, each representing
10//! different kinds of blocks, and different kinds of ownership.
11//!
12//! | `block2` type                            | Equivalent Rust type  |
13//! | ---------------------------------------- | --------------------- |
14//! | `&Block<dyn Fn() + 'a>`                  | `&dyn Fn() + 'a`      |
15//! | `RcBlock<dyn Fn() + 'a>`                 | `Arc<dyn Fn() + 'a>`  |
16//! | `StackBlock<'a, (), (), impl Fn() + 'a>` | `impl Fn() + 'a`      |
17//! | `GlobalBlock<dyn Fn()>`                  | [`fn` item]           |
18//!
19//! For more information on the specifics of the block implementation, see the
20//! [C language specification][lang] and the [ABI specification][ABI].
21//!
22//! [lang]: https://clang.llvm.org/docs/BlockLanguageSpec.html
23//! [ABI]: http://clang.llvm.org/docs/Block-ABI-Apple.html
24//! [`fn` item]: https://doc.rust-lang.org/reference/types/function-item.html
25//!
26//!
27//! ## External functions using blocks
28//!
29//! To declare external functions or methods that takes blocks, use
30//! `&Block<dyn Fn(Params) -> R>` or `Option<&Block<dyn Fn(Args) -> R>>`,
31//! where `Params` is the parameter types, and `R` is the return type.
32//!
33//! In the next few examples, we're going to work with a function
34//! `check_addition`, that takes as parameter a block that adds two integers,
35//! and checks that the addition is correct.
36//!
37//! Such a function could be written in C like in the following.
38//!
39//! ```objc
40//! #include <cassert>
41//! #include <stdint.h>
42//! #include <Block.h>
43//!
44//! void check_addition(int32_t (^block)(int32_t, int32_t)) {
45//!     assert(block(5, 8) == 13);
46//! }
47//! ```
48//!
49//! An `extern "C" { ... }` declaration for that function would then be:
50//!
51//! ```
52//! use block2::Block;
53//!
54//! extern "C" {
55//!     fn check_addition(block: &Block<dyn Fn(i32, i32) -> i32>);
56//! }
57//! ```
58//!
59//! This can similarly be done inside external methods declared with
60//! [`objc2::extern_methods!`].
61//!
62//! ```
63//! use block2::Block;
64//! use objc2::extern_methods;
65//! #
66//! # use objc2::ClassType;
67//! # objc2::extern_class!(
68//! #     struct MyClass;
69//! #
70//! #     unsafe impl ClassType for MyClass {
71//! #         type Super = objc2::runtime::NSObject;
72//! #         type Mutability = objc2::mutability::InteriorMutable;
73//! #         const NAME: &'static str = "NSObject";
74//! #     }
75//! # );
76//!
77//! extern_methods!(
78//!     unsafe impl MyClass {
79//!         #[method(checkAddition:)]
80//!         pub fn checkAddition(&self, block: &Block<dyn Fn(i32, i32) -> i32>);
81//!     }
82//! );
83//! ```
84//!
85//! If the function/method allowed passing `NULL` blocks, the type would be
86//! `Option<&Block<dyn Fn(i32, i32) -> i32>>` instead.
87//!
88//!
89//! ## Invoking blocks
90//!
91//! We can also define the external function in Rust, and expose it to
92//! Objective-C. To do this, we can use [`Block::call`] to invoke the block
93//! inside the function.
94//!
95//! ```
96//! use block2::Block;
97//!
98//! #[no_mangle]
99//! extern "C" fn check_addition(block: &Block<dyn Fn(i32, i32) -> i32>) {
100//!     assert_eq!(block.call((5, 8)), 13);
101//! }
102//! ```
103//!
104//! Note the extra parentheses in the `call` method, since the arguments must
105//! be passed as a tuple.
106//!
107//!
108//! ## Creating blocks
109//!
110//! Creating a block to pass to Objective-C can be done with [`RcBlock`] or
111//! [`StackBlock`], depending on if you want to move the block to the heap,
112//! or let the callee decide if it needs to do that.
113//!
114//! To call such a function / method, we could create a new block from a
115//! closure using [`RcBlock::new`].
116//!
117//! ```
118//! use block2::RcBlock;
119//! #
120//! # extern "C" fn check_addition(block: &block2::Block<dyn Fn(i32, i32) -> i32>) {
121//! #     assert_eq!(block.call((5, 8)), 13);
122//! # }
123//!
124//! let block = RcBlock::new(|a, b| a + b);
125//! check_addition(&block);
126//! ```
127//!
128//! This creates the block on the heap. If the external function you're
129//! calling is not going to copy the block, it may be more performant if you
130//! construct a [`StackBlock`] directly, using [`StackBlock::new`].
131//!
132//! Note that this requires that the closure is [`Clone`], as the external
133//! code is allowed to copy the block to the heap in the future.
134//!
135//! ```
136//! use block2::StackBlock;
137//! #
138//! # extern "C" fn check_addition(block: &block2::Block<dyn Fn(i32, i32) -> i32>) {
139//! #     assert_eq!(block.call((5, 8)), 13);
140//! # }
141//!
142//! let block = StackBlock::new(|a, b| a + b);
143//! check_addition(&block);
144//! ```
145//!
146//! As an optimization, if your closure doesn't capture any variables (as in
147//! the above examples), you can use the [`global_block!`] macro to create a
148//! static block.
149//!
150//! ```
151//! use block2::global_block;
152//! #
153//! # extern "C" fn check_addition(block: &block2::Block<dyn Fn(i32, i32) -> i32>) {
154//! #     assert_eq!(block.call((5, 8)), 13);
155//! # }
156//!
157//! global_block! {
158//!     static BLOCK = |a: i32, b: i32| -> i32 {
159//!         a + b
160//!     };
161//! }
162//!
163//! check_addition(&BLOCK);
164//! ```
165//!
166//!
167//! ## Lifetimes
168//!
169//! When dealing with blocks, there can be quite a few lifetimes to keep in
170//! mind.
171//!
172//! The most important one is the lifetime of the block's data, i.e. the
173//! lifetime of the data in the closure contained in the block. This lifetime
174//! can be specified as `'f` in `&Block<dyn Fn() + 'f>`.
175//!
176//! Note that `&Block<dyn Fn()>`, without any lifetime specifier, can be a bit
177//! confusing, as the default depends on where it is typed. In function/method
178//! signatures, it defaults to `'static`, but as the type of e.g. a `let`
179//! binding, the lifetime may be inferred to be something smaller, see [the
180//! reference][ref-dyn-lifetime] for details. If in doubt, either add a
181//! `+ 'static` or `+ '_` to force an escaping or non-escaping block.
182//!
183//! Another lifetime is the lifetime of the currently held pointer, i.e. `'b`
184//! in `&'b Block<dyn Fn()>`. This lifetime can be safely extended using
185//! [`Block::copy`], so should prove to be little trouble (of course the
186//! lifetime still can't be extended past the lifetime of the captured data
187//! above).
188//!
189//! Finally, the block's parameter and return types can also contain
190//! lifetimes, as `'a` and `'r` in `&Block<dyn Fn(&'a i32) -> &'r u32>`.
191//! Unfortunately, these lifetimes are quite problematic and unsupported at
192//! the moment, due to Rust trait limitations regarding higher-ranked trait
193//! bounds. If you run into problems with this in a block that takes or
194//! returns a reference, consider using the ABI-compatible `NonNull<T>`, or
195//! transmute to a `'static` lifetime.
196//!
197//! [ref-dyn-lifetime]: https://doc.rust-lang.org/reference/lifetime-elision.html#default-trait-object-lifetimes
198//!
199//!
200//! ## Thread safety
201//!
202//! Thread-safe blocks are not yet representable in `block2`, and as such any
203//! function that requires a thread-safe block must be marked `unsafe`.
204//!
205//!
206//! ## Mutability
207//!
208//! Blocks are generally assumed to be shareable, and as such can only very
209//! rarely be made mutable. In particular, there is no good way to prevent
210//! re-entrancy.
211//!
212//! You will likely have to use interior mutability instead.
213//!
214//!
215//! ## Specifying a runtime
216//!
217//! Different runtime implementations exist and act in slightly different ways
218//! (and have several different helper functions), the most important aspect
219//! being that the libraries are named differently, so we must take that into
220//! account when linking.
221//!
222//! You can choose the desired runtime by using the relevant cargo feature
223//! flags, see the following sections (you might have to disable the default
224//! `"apple"` feature first).
225//!
226//!
227//! ### Apple's [`libclosure`](https://github.com/apple-oss-distributions/libclosure)
228//!
229//! - Feature flag: `apple`.
230//!
231//! This is the most common and most sophisticated runtime, and it has quite a
232//! lot more features than the specification mandates.
233//!
234//! The minimum required operating system versions are as follows (though in
235//! practice Rust itself requires higher versions than this):
236//!
237//! - macOS: `10.6`
238//! - iOS/iPadOS: `3.2`
239//! - tvOS: `1.0`
240//! - watchOS: `1.0`
241//!
242//! **This is used by default**, so you do not need to specify a runtime if
243//! you're using this crate on of these platforms.
244//!
245//!
246//! ### LLVM `compiler-rt`'s [`libBlocksRuntime`](https://github.com/llvm/llvm-project/tree/release/13.x/compiler-rt/lib/BlocksRuntime)
247//!
248//! - Feature flag: `compiler-rt`.
249//!
250//! This is a copy of Apple's older (around macOS 10.6) runtime, and is now
251//! used in [Swift's `libdispatch`] and [Swift's Foundation] as well.
252//!
253//! The runtime and associated headers can be installed on many Linux systems
254//! with the `libblocksruntime-dev` package.
255//!
256//! Using this runtime probably won't work together with `objc2` crate.
257//!
258//! [Swift's `libdispatch`]: https://github.com/apple/swift-corelibs-libdispatch/tree/swift-5.5.1-RELEASE/src/BlocksRuntime
259//! [Swift's Foundation]: https://github.com/apple/swift-corelibs-foundation/tree/swift-5.5.1-RELEASE/Sources/BlocksRuntime
260//!
261//!
262//! ### GNUStep's [`libobjc2`](https://github.com/gnustep/libobjc2)
263//!
264//! - Feature flag: `gnustep-1-7`, `gnustep-1-8`, `gnustep-1-9`, `gnustep-2-0`
265//!   and `gnustep-2-1` depending on the version you're using.
266//!
267//! GNUStep is a bit odd, because it bundles blocks support into its
268//! Objective-C runtime. This means we have to link to `libobjc`, and this is
269//! done by depending on the `objc2` crate. A bit unorthodox, yes, but it
270//! works.
271//!
272//! Sources:
273//!
274//! - [`Block.h`](https://github.com/gnustep/libobjc2/blob/v2.1/objc/blocks_runtime.h)
275//! - [`Block_private.h`](https://github.com/gnustep/libobjc2/blob/v2.1/objc/blocks_private.h)
276//!
277//!
278//! ### Microsoft's [`WinObjC`](https://github.com/microsoft/WinObjC)
279//!
280//! - Feature flag: `unstable-winobjc`.
281//!
282//! **Unstable: Hasn't been tested on Windows yet!**
283//!
284//! [A fork](https://github.com/microsoft/libobjc2) based on GNUStep's `libobjc2`
285//! version 1.8.
286//!
287//!
288//! ### [`ObjFW`](https://github.com/ObjFW/ObjFW)
289//!
290//! - Feature flag: `unstable-objfw`.
291//!
292//! **Unstable: Doesn't work yet!**
293//!
294//!
295//! ## C Compiler configuration
296//!
297//! To our knowledge, only Clang supports blocks. To compile C or Objective-C
298//! sources using these features, you should set [the `-fblocks` flag][flag].
299//!
300//! [flag]: https://clang.llvm.org/docs/ClangCommandLineReference.html#cmdoption-clang-fblocks
