7/// Send a message to an object or class.
8///
9/// This is wildly `unsafe`, even more so than sending messages in
10/// Objective-C, because this macro can't inspect header files to see the
11/// expected types, and because Rust has more safety invariants to uphold.
12/// Make sure to review the safety section below!
13///
14/// The recommended way of using this macro is by defining a wrapper function:
15///
16/// ```
17/// # use std::ffi::{c_int, c_char};
18/// # use objc2::msg_send;
19/// # use objc2::runtime::NSObject;
20/// unsafe fn do_something(obj: &NSObject, arg: c_int) -> *const c_char {
21///     msg_send![obj, doSomething: arg]
22/// }
23/// ```
24///
25/// This way we are clearly communicating to Rust that: The method
26/// `doSomething:` works with a shared reference to the object. It takes a
27/// C-style signed integer, and returns a pointer to what is probably a
28/// C-compatible string. Now it's much, _much_ easier to make a safe
29/// abstraction around this!
30///
31/// The [`extern_methods!`] macro can help with coding this pattern.
32///
33/// [`extern_methods!`]: crate::extern_methods
34///
35///
36/// # Memory management
37///
38/// If an Objective-C method returns `id`, `NSObject*`, or similar object
39/// pointers, you should use [`Retained<T>`] on the Rust side, or
40/// `Option<Retained<T>>` if the pointer is nullable.
41///
42/// This is necessary because object pointers in Objective-C have certain
43/// rules for when they should be retained and released across function calls.
44///
45/// [`Retained<T>`]: crate::rc::Retained
46///
47///
48/// ## A little history
49///
50/// Objective-C's type system is... limited, so you can't tell without
51/// consulting the documentation who is responsible for releasing an object.
52/// To remedy this problem, Apple/Cocoa introduced (approximately) the
53/// following rule:
54///
55/// The caller is responsible for releasing objects return from methods that
56/// begin with `new`, `alloc`, `copy`, `mutableCopy` or `init`, and method
57/// that begins with `init` takes ownership of the receiver. See [Cocoa's
58/// Memory Management Policy][mmRules] for a user-friendly introduction to
59/// this concept.
60///
61/// In the past, users had to do `retain` and `release` calls themselves to
62/// properly follow these rules. To avoid the memory management problems
63/// associated with manual stuff like that, they [introduced "ARC"][arc-rel],
64/// which codifies the rules as part of the language, and inserts the required
65/// `retain` and `release` calls automatically.
66///
67/// Returning a `*const T` pointer is similar to pre-ARC; you have to know
68/// when to retain and when to release an object. Returning `Retained` is
69/// similar to ARC; the rules are simple enough that we can do them
70/// automatically!
71///
72/// [mmRules]: https://developer.apple.com/library/archive/documentation/Cocoa/Conceptual/MemoryMgmt/Articles/mmRules.html#//apple_ref/doc/uid/20000994-SW1
73/// [arc-rel]: https://developer.apple.com/library/archive/releasenotes/ObjectiveC/RN-TransitioningToARC/Introduction/Introduction.html#//apple_ref/doc/uid/TP40011226
74///
75///
76/// # Specification
77///
78/// The syntax is somewhat similar to the message syntax in Objective-C,
79/// except with a comma between arguments. Eliding the comma is possible, but
80/// deprecated, and may be removed in a future version of `objc2`.
81///
82/// The first expression, know as the "receiver", can be any type that
83/// implements [`MessageReceiver`], like a reference or a pointer to an
84/// object. Additionally, it can even be a reference to an [`Retained`]
85/// containing an object.
86///
87/// The expression can be wrapped in `super`, with an optional superclass
88/// as the second argument. If no specific superclass is specified, the
89/// direct superclass is retrieved from [`ClassType`].
90///
91/// All arguments, as well as the return type, must implement [`Encode`] (bar
92/// the exceptions below).
93///
94/// If the last argument is the special marker `_`, the macro will return a
95/// `Result<_, Retained<E>>`, see below.
96///
97/// This macro roughly translates into a call to [`sel!`], and afterwards a
98/// fully qualified call to [`MessageReceiver::send_message`]. Note that this
99/// means that auto-dereferencing of the receiver is not supported, and that
100/// the receiver is consumed. You may encounter a little trouble with `&mut`
101/// references, try refactoring into a separate method or reborrowing the
102/// reference.
103///
104/// Variadic arguments are currently not supported.
105///
106/// [`MessageReceiver`]: crate::runtime::MessageReceiver
107/// [`Retained`]: crate::rc::Retained
108/// [`ClassType`]: crate::ClassType
109/// [`Encode`]: crate::Encode
110/// [`sel!`]: crate::sel
111/// [`MessageReceiver::send_message`]: crate::runtime::MessageReceiver::send_message
112///
113///
114/// ## Memory management details
115///
116/// The accepted receiver and return types, and how we handle them, differ
117/// depending on which, if any, of the [recognized selector
118/// families][sel-families] the selector belongs to:
119///
120/// - The `new` family: The receiver may be anything that implements
121///   [`MessageReceiver`] (though often you'll want to use `&AnyClass`). The
122///   return type is a generic `Retained<T>` or `Option<Retained<T>>`.
123///
124/// - The `alloc` family: The receiver must be `&AnyClass`, and the return
125///   type is a generic `Allocated<T>`.
126///
127/// - The `init` family: The receiver must be `Allocated<T>` as returned from
128///   `alloc`, or if sending messages to the superclass, it must be
129///   `PartialInit<T>`.
130///
131///   The receiver is consumed, and a the now-initialized `Retained<T>` or
132///   `Option<Retained<T>>` (with the same `T`) is returned.
133///
134/// - The `copy` family: The receiver may be anything that implements
135///   [`MessageReceiver`] and the return type is a generic `Retained<T>` or
136///   `Option<Retained<T>>`.
137///
138/// - The `mutableCopy` family: Same as the `copy` family.
139///
140/// - No family: The receiver may be anything that implements
141///   [`MessageReceiver`]. The result is retained using
142///   [`Retained::retain_autoreleased`], and a generic `Retained<T>` or
143///   `Option<Retained<T>>` is returned. This retain is in most cases faster
144///   than using autorelease pools!
145///
146/// See [the clang documentation][arc-retainable] for the precise
147/// specification of Objective-C's ownership rules.
148///
149/// As you may have noticed, the return type is usually either `Retained` or
150/// `Option<Retained>`. Internally, the return type is always
151/// `Option<Retained>` (for example: almost all `new` methods can fail if the
152/// allocation failed), but for convenience, if the return type is
153/// `Retained<T>`, this macro will automatically unwrap the object, or panic
154/// with an error message if it couldn't be retrieved.
155///
156/// As a special case, if the last argument is the marker `_`, the macro will
157/// return a `Result<Retained<T>, Retained<E>>`, see below.
158///
159/// The `retain`, `release` and `autorelease` selectors are not supported, use
160/// [`Retained::retain`], [`Retained::drop`] and [`Retained::autorelease_ptr`]
161/// for that.
162///
163/// [sel-families]: https://clang.llvm.org/docs/AutomaticReferenceCounting.html#arc-method-families
164/// [`MessageReceiver`]: crate::runtime::MessageReceiver
165/// [`Retained::retain_autoreleased`]: crate::rc::Retained::retain_autoreleased
166/// [arc-retainable]: https://clang.llvm.org/docs/AutomaticReferenceCounting.html#retainable-object-pointers-as-operands-and-arguments
167/// [`Retained::retain`]: crate::rc::Retained::retain
168/// [`Retained::drop`]: crate::rc::Retained::drop
169/// [`Retained::autorelease_ptr`]: crate::rc::Retained::autorelease_ptr
170///
171///
172/// # `bool` handling
173///
174/// Objective-C's `BOOL` is slightly different from Rust's [`bool`], and hence
175/// a conversion step must be performed before using it. This is _very_ easy
176/// to forget (because it'll happen to work in _most_ cases), so this macro
177/// does the conversion step automatically whenever an argument or the return
178/// type is `bool`.
179///
180/// That means that any Objective-C method that take or return `BOOL` can be
181/// translated to use `bool` on the Rust side.
182///
183/// If you want to handle the conversion explicitly, or the Objective-C method
184/// expects e.g. a pointer to a `BOOL`, use [`runtime::Bool`] instead.
185///
186/// [`runtime::Bool`]: crate::runtime::Bool
187///
188///
189/// # `CStr` handling
190///
191/// Since the ABI of [`CStr`][std::ffi::CStr] isn't yet stable, `&CStr` and
192/// `Option<&CStr>` are converted to/from `*const c_char`.
193///
194///
195/// # Out-parameters
196///
197/// Parameters like `NSString**` in Objective-C are passed by "writeback",
198/// which means that the callee autoreleases any value that they may write
199/// into the parameter.
200///
201/// This macro has support for passing such parameters using the following
202/// types:
203/// - `&mut Retained<_>`
204/// - `Option<&mut Retained<_>>`
205/// - `&mut Option<Retained<_>>`,
206/// - `Option<&mut Option<Retained<_>>>`
207///
208/// Beware with the first two, since they will cause undefined behaviour if
209/// the method overwrites the value with `nil`.
210///
211/// See [clang's documentation][clang-out-params] for more details.
212///
213/// [clang-out-params]: https://clang.llvm.org/docs/AutomaticReferenceCounting.html#passing-to-an-out-parameter-by-writeback
214///
215///
216/// # Errors
217///
218/// The most common place you'll see out-parameters is as `NSError**` the last
219/// parameter, which is used to communicate errors to the caller, see [Error
220/// Handling Programming Guide For Cocoa][cocoa-error].
221///
222/// Similar to Swift's [importing of error parameters][swift-error], this
223/// macro supports an even more convenient version than the out-parameter
224/// support, which transforms methods whose last parameter is `NSError**` into
225/// the Rust equivalent, the [`Result`] type.
226///
227/// In particular, if you make the last argument the special marker `_`, then
228/// the macro will return a `Result<R, Retained<E>>`. The error type `E` must
229/// be either [`NSObject`] or `objc2_foundation::NSError`.
230///
231/// The success type `R` must be either `()` or `Retained<T>`.
232///
233/// At runtime, we create the temporary error variable for you on the stack
234/// and send it as the out-parameter to the method. If the method then returns
235/// `NO`/`false`, or in the case of an object pointer, `NULL`, the error
236/// variable is loaded and returned in [`Err`].
237///
238/// [cocoa-error]: https://developer.apple.com/library/archive/documentation/Cocoa/Conceptual/ErrorHandlingCocoa/ErrorHandling/ErrorHandling.html
239/// [swift-error]: https://developer.apple.com/documentation/swift/about-imported-cocoa-error-parameters
240/// [`NSObject`]: crate::runtime::NSObject
241///
242///
243/// # Panics
244///
245/// Unwinds if the underlying method throws and exception. If the
246/// `"catch-all"` Cargo feature is enabled, the Objective-C exception is
247/// converted into a Rust panic, with potentially a bit better stack trace.
248///
249/// Finally, panics if the return type is specified as `Retained<_>`, but the
250/// method actually returned NULL. If this happens, you should change the
251/// signature to instead return `Option<Retained<_>>` to handle the error
252/// yourself.
253///
254///
255/// ## Type verification
256///
257/// To make message sending safer, all arguments and return values for
258/// messages must implement [`encode::Encode`]. This allows the Rust compiler
259/// to prevent you from passing e.g. a [`Vec`] into Objective-C, which would
260/// both be UB and leak the vector.
261///
262/// When `debug_assertions` are enabled, this macro will check the encoding of
263/// the given arguments and return every time you send a message, and will
264/// panic if they are not equivalent.
265///
266/// This is not a perfect solution for ensuring safety (some Rust types have
267/// the same Objective-C encoding, but are not equivalent, such as `&T` and
268/// `*const T`), but it gets us much closer to it!
269///
270/// This behaviour can be tweaked with the `"relax-void-encoding"`,
271/// `"relax-sign-encoding"` or `"disable-encoding-assertions"` Cargo feature
272/// flags if it is causing you trouble.
273///
274/// [`encode::Encode`]: crate::encode::Encode
275/// [`Vec`]: std::vec::Vec
276///
277///
278/// # Safety
279///
280/// Similar to defining and calling an `extern` function in a foreign function
281/// interface. In particular, you must uphold the following requirements:
282///
283/// 1. The selector corresponds to a valid method that is available on the
284///    receiver.
285///
286/// 2. The argument types match what the receiver excepts for this selector.
287///
288/// 3. The return type match what the receiver returns for this selector.
289///
290/// 4. The call must not violate Rust's mutability rules, for example if
291///    passing an `&T`, the Objective-C method must not mutate the variable
292///    (except if the variable is inside [`std::cell::UnsafeCell`] or
293///    derivatives).
294///
295/// 5. If the receiver is a raw pointer it must be valid (aligned,
296///    dereferenceable, initialized and so on). Messages to `null` pointers
297///    are allowed (though heavily discouraged), but _only_ if the return type
298///    itself is a pointer.
299///
300/// 6. You must uphold any additional safety requirements (explicit and
301///    implicit) that the method has. For example:
302///    - Methods that take pointers usually require that the pointer is valid,
303///      and sometimes non-null.
304///    - Sometimes, a method may only be called on the main thread.
305///    - The lifetime of returned pointers usually follows certain rules, and
306///      may not be valid outside of an [`autoreleasepool`] (returning
307///      `Retained` usually helps with these cases).
308///
309/// 7. Each out-parameter must have the correct nullability, and the method
310///    must not have any attributes that changes the how it handles memory
311///    management for these.
312///
313/// 8. If using the automatic memory management facilities of this macro, the
314///    method must not have any attributes such as `objc_method_family`,
315///    `ns_returns_retained`, `ns_consumed` that changes the how it handles
316///    memory management.
317///
318/// 8. TODO: Maybe more?
319///
320/// [`autoreleasepool`]: crate::rc::autoreleasepool
321///
322///
323/// # Examples
324///
325/// Interacting with [`NSURLComponents`], [`NSString`] and [`NSNumber`].
326///
327/// [`NSURLComponents`]: https://developer.apple.com/documentation/foundation/nsurlcomponents?language=objc
328/// [`NSString`]: https://developer.apple.com/documentation/foundation/nsstring?language=objc
329/// [`NSNumber`]: https://developer.apple.com/documentation/foundation/nsnumber?language=objc
330///
331/// ```
332/// use objc2::rc::Retained;
333/// use objc2::{msg_send, ClassType};
334/// use objc2::runtime::NSObject;
335///
336/// // Declare stub types for the three classes we'll be working with.
337/// objc2::extern_class!(
338///     #[unsafe(super(NSObject))] // + NSValue
339///     struct NSNumber;
340/// );
341/// objc2::extern_class!(
342///     #[unsafe(super(NSObject))]
343///     struct NSURLComponents;
344/// );
345/// objc2::extern_class!(
346///     #[unsafe(super(NSObject))]
347///     struct NSString;
348/// );
349///
350///
351/// // Create an empty `NSURLComponents` by calling the class method `new`.
352/// let components: Retained<NSURLComponents> = unsafe {
353///     //          ^^^^^^^^^^^^^^^^^^^^^^^^^ the return type, a memory-managed
354///     //                                    `NSURLComponents` instance
355///     //
356///     msg_send![NSURLComponents::class(), new]
357///     //        ------------------------  ^^^ the selector `new`
358///     //        |
359///     //        the receiver, in this case the class itself
360/// };
361///
362///
363/// // Create a new `NSNumber` from an integer.
364/// let port: Retained<NSNumber> = unsafe {
365///     msg_send![NSNumber::class(), numberWithInt: 8080i32]
366///     //                           -------------- ^^^^^^^ the argument to the method
367///     //                           |
368///     //                           the selector `numberWithInt:`
369///     //
370///     // Note how we must fully specify the argument as `8080i32` instead of just `8080`.
371/// };
372///
373///
374/// // Set the port property of the URL.
375/// let _: () = unsafe { msg_send![&components, setPort: &*port] };
376/// //     --                                   -------- ^^^^^^ the port is deref'd to
377/// //     |                                    |               become the correct type
378/// //     |                                    |
379/// //     |                                    the selector `setPort:` is derived
380/// //     |                                    from the property name `port`.
381/// //     |
382/// //     return type (i.e. nothing / void)
383/// //
384/// // Note that even return types of `void` must be explicitly specified as `()`.
385///
386///
387/// // Set the `host` property of the URL.
388/// let host: Retained<NSString> = unsafe {
389///     msg_send![NSString::class(), stringWithUTF8String: c"example.com".as_ptr()]
390/// };
391/// let _: () = unsafe { msg_send![&components, setHost: &*host] };
392///
393///
394/// // Set the `scheme` property of the URL.
395/// let scheme: Retained<NSString> = unsafe {
396///     msg_send![NSString::class(), stringWithUTF8String: c"http".as_ptr()]
397/// };
398/// let _: () = unsafe { msg_send![&components, setScheme: &*scheme] };
399///
400///
401/// // Get the combined URL in string form.
402/// let string: Option<Retained<NSString>> = unsafe { msg_send![&components, string] };
403/// //          ^^^^^^ the method can return NULL, so we specify an option here
404/// let string = string.unwrap();
405///
406/// // Convert the `NSString` to a Rust string.
407/// // NOTE: This has issues with interior NUL characters.
408/// // `objc2_foundation::NSString` handles this correctly.
409/// let bytes: *const std::ffi::c_char = unsafe { msg_send![&string, UTF8String] };
410/// let c_str = unsafe { std::ffi::CStr::from_ptr(bytes) };
411///
412/// assert_eq!(c_str, c"http://example.com:8080");
413/// ```
414///
415/// The example above uses only `msg_send!` for demonstration purposes; note
416/// that usually the interface you seek is already present in [the framework
417/// crates] and then the equivalent code can be as simple as:
418///
419/// [the framework crates]: crate::topics::frameworks_list
420///
421/// ```ignore
422/// use objc2_foundation::{NSNumber, NSString, NSURLComponents};
423///
424/// let components = NSURLComponents::new();
425/// components.setPort(Some(&NSNumber::new_i32(8080)));
426/// components.setHost(Some(&NSString::from_str("example.com")));
427/// components.setScheme(Some(&NSString::from_str("http")));
428/// let string = components.string();
429///
430/// assert_eq!(string.unwrap().to_string(), "http://example.com:8080");
431/// ```
432///
433/// Sending messages to the superclass of an object.
434///
435/// ```no_run
436/// use objc2::runtime::NSObject;
437/// use objc2::{msg_send, ClassType};
438/// #
439/// # objc2::define_class!(
440/// #     #[unsafe(super(NSObject))]
441/// #     struct MyObject;
442/// # );
443/// #
444/// # let obj: objc2::rc::Retained<MyObject> = todo!();
445///
446/// // Call `someMethod` on the direct super class.
447/// let _: () = unsafe { msg_send![super(&obj), someMethod] };
448///
449/// // Or lower-level, a method on a specific superclass.
450/// let superclass = NSObject::class();
451/// let arg3: u32 = unsafe { msg_send![super(&obj, superclass), getArg3] };
452/// ```
453///
454/// Sending a message with automatic error handling.
455///
456/// ```no_run
457/// use objc2::msg_send;
458/// use objc2::rc::Retained;
459/// # #[cfg(requires_foundation)]
460/// use objc2_foundation::{NSBundle, NSError};
461/// # use objc2::runtime::NSObject as NSBundle;
462/// # use objc2::runtime::NSObject as NSError;
463///
464/// # #[cfg(requires_foundation)]
465/// let bundle = NSBundle::mainBundle();
466/// # let bundle = NSBundle::new();
467///
468/// let res: Result<(), Retained<NSError>> = unsafe {
469///     //          --  -------- ^^^^^^^ must be NSError or NSObject
470///     //          |   |
471///     //          |   always retained
472///     //          |
473///     //          `()` means that the method returns `bool`, we check
474///     //          that and return success if `true`, an error if `false`
475///     //
476///     msg_send![&bundle, preflightAndReturnError: _]
477///     //                                          ^ activate error handling
478/// };
479/// ```
480///
481/// Sending a message with an out parameter _and_ automatic error handling.
482///
483/// ```no_run
484/// use objc2::msg_send;
485/// use objc2::rc::Retained;
486///
487/// # type NSFileManager = objc2::runtime::NSObject;
488/// # type NSURL = objc2::runtime::NSObject;
489/// # type NSError = objc2::runtime::NSObject;
490/// let obj: &NSFileManager;
491/// # obj = todo!();
492/// let url: &NSURL;
493/// # url = todo!();
494/// let mut result_url: Option<Retained<NSURL>> = None;
495/// let _: () = unsafe {
496///     msg_send![
497///         obj,
498///         trashItemAtURL: url,
499///         resultingItemURL: Some(&mut result_url),
500///         error: _
501///     ]?
502/// //   ^ is possible on error-returning methods, if the return type is specified
503/// };
504///
505/// // Use `result_url` here
506///
507/// # Ok::<(), Retained<NSError>>(())
508/// ```
509///
510/// Attempt to do an invalid message send. This is undefined behaviour, but
511/// will panic with `debug_assertions` enabled.
512///
513/// ```should_panic
514/// use objc2::msg_send;
515/// use objc2::runtime::NSObject;
516///
517/// let obj = NSObject::new();
518///
519/// // Wrong return type - this is UB!
520/// //
521/// // But it will be caught with `debug_assertions` enabled, stating that
522/// // the return type's encoding is not correct.
523/// let hash: f32 = unsafe { msg_send![&obj, hash] };
524/// #
525/// # panic!("does not panic in release mode, so for testing we make it!");
526/// ```
527#[macro_export]
528macro_rules! msg_send {
529    [super($obj:expr), $($selector_and_arguments:tt)+] => {
530        $crate::__msg_send_parse! {
531            ()
532            ()
533            ($($selector_and_arguments)+)
534
535            (MsgSendSuperError::send_super_message_static_error)
536            (MsgSendSuper::send_super_message_static)
537
538            ($crate::__msg_send_helper)
539            ($obj)
540        }
541    };
542    [super($obj:expr, $superclass:expr), $($selector_and_arguments:tt)+] => {
543        $crate::__msg_send_parse! {
544            ()
545            ()
546            ($($selector_and_arguments)+)
547
548            (MsgSendSuperError::send_super_message_error)
549            (MsgSendSuper::send_super_message)
550
551            ($crate::__msg_send_helper)
552            ($obj, $superclass)
553        }
554    };
555    [$obj:expr, $($selector_and_arguments:tt)+] => {
556        $crate::__msg_send_parse! {
557            ()
558            ()
559            ($($selector_and_arguments)+)
560
561            (MsgSendError::send_message_error)
562            (MsgSend::send_message)
563
564            ($crate::__msg_send_helper)
565            ($obj)
566        }
567    };
568}

591#[doc(hidden)]
592#[macro_export]
593macro_rules! __msg_send_helper {
594    {
595        ($($fn_args:tt)+)
596        ($trait:ident :: $fn:ident)
597        ($($selector:tt)*)
598        ($($argument:expr,)*)
599    } => ({
600        // Assign to intermediary variable for better UI, and to prevent
601        // miscompilation on older Rust versions (TODO: Which ones?).
602        //
603        // Note: This can be accessed from any expression in `fn_args` and
604        // `argument` - we won't (yet) bother with preventing that though.
605        let result;
606
607        // Always add trailing comma after each argument, so that we get a
608        // 1-tuple if there is only one.
609        //
610        // And use `::<_, _>` for better UI.
611        result = <$crate::__method_family!(() ($($selector)*)) as $crate::__macros::$trait<_, _>>::$fn(
612            $($fn_args)+,
613            $crate::sel!($($selector)*),
614            ($($argument,)*),
615        );
616        result
617    });
